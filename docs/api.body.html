(1.0 WIP)Transport

  • All content-modifying actions (POST, PUT, DELETE) require an _authentication_token argument; since this is universal, it isn't mentioned in the docs for each action separately. The token can be found by calling /account/signin.json or /account/signin.xml
  • Cookies are used to keep track of logged in sessions. Your client may be sent additional cookies while using the API. These should be stored and retreived as normal cookies.
  • PUT and DELETE can be emulated with POST, having the form variable "_method" set to PUT or DELETE respectively
  • HTTP status codes are used:
    • 200 - generic success
    • 201 - content created
    • 400 - generic user error (commonly missing required data when submitting a form)
    • 403 - permission denied
    • 404 - content/member not found
    • 500 - generic server error
    • 501 - not implemented
    • 502 - back-end server is down for maintenance

(1.0 WIP)Data formats

Formats can be added to the end of the URL; for instance while /contents/42 will show you content #42 in a browser-friendly way (HTML), a standalone app may prefer /contentss/42.json or /contents/42.xml

Most examples in this documentation will be given in JSON format.

Format list:

  • JSON
  • XML
  • Frag
  • RSS

JSON

JSON data format: 

    {
        "status" : "ok" / "error" / "invalid",                 (required)
        "message": "string to show in app status bar / popup", (required; can be null if status=ok)
        "data"   : {                                           (required; blank dictionary if there is no data)
            "foo": "bar", # function-specific data
            "baz": "qux",
            ...
        }
    }

JSON list format: 

    {
        "status" : "ok",
        "message": null,
        "data"   : {
            "list": {
                "items": [
                    {...},
                    {...},
                    {...}
                ],
                "type"  : "content" / "member" / "message",
                "limit" : 20,
                "offset":  0,
                "count" :  3
            }
        }
    }

XML

The same as JSON, just in XML.

Frag

Fragments are chunks of HTML data with marked up with class's for use with injecting content directly into exisiting webpages. The developer can then style this injected HTML simply with standard CSS rules

WIP NOTE: Currently .frag returns HTML specific to the Civiciboom site and is not generic enough for mainstream use. Work is currently underway to make the .frag a generic tool for developers. (tentativly May 2011)

RSS

Objects and Lists can be returned as RSS feeds

Some of the fields are not relevenet for an RSS output. The list is fields returned in the RSS ofen a subset of the JSON/XML output.

It is part of the RSS standard that clients can sort items by date. Some objects may return lists of items that are not in date order.

Hint: Common use of RSS 'content' lists: By default media attachments are not included in the RSS return but can be enabled by including the paramiter 'include_fields=attachments'

(1.0 WIP)List Controls

API calls masked as 'list' will return lists of objects. These lists can be customised by passing common control parmiters.

  • list_type: default='default'

    Permitted Values

    • 'empty' = no fields (to be used with include_fields to build a custom object)
    • 'default' = object fields (usually used for list as no links to other objects are present)
    • 'full' = object + links (links to sub objects or lists are followed e.g a contents creator field will contain a sub object member). full field lists often include fields that are expensive to process and are normally unnessisary for summary lists.

  • 'exclude_fields': ommit fields from the return object default=''

    e.g "exclude_fields=avatar_url,location_home" will ommit the fields avatar_url and location_home form the returned list.

  • 'include_fields': fields to include in the return object deafult=''

    e.g 'include_fields=followers,username,name,webpage,content_public'.

    e.g 'include_fields=attachments'

    Often used in conjection with 'list_type=empty' to build a custom return object. Often used to add addtional linked fields such as

  • sort: (not implmented - planned for June 2011)

    comma separted list of col names e.g rating,creator,-update_date (- denotes alternate sorting)

  • limit: limit items returned default='20'

    e.g 10 - only return 10 items max

    e.g 0 - return all items with no maximum limit

  • offset: define where in the list to return default='0'

    e.g 30 - start the listing at item 30

(1.0 WIP)Validation

If an invalid request is made the following structure will be returned. 

    {
        "status" : "invalid",
        "message": "description of invalid state", 
        "data"   : {
            "foo": "bar",
            "baz": "qux",
            ...
            "invalid": {
                "foo": "description of validation error"
            }
        }
    }

Invalid feilds are listed in the 'invalid' dictonary with a text description of why the validation failed

(1.0 WIP)Concept of 'me'

Anywhere a username or user_id are present in the API, you may submit the value 'me' to refer to the currently logged in user e.g:

  • members/me/content
  • members?followed_by=me
  • contents?boomed_by=me
